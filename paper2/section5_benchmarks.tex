\section{Empirical Validation}
\label{sec:bench}

\subsection{Experimental Setup}

All measurements were collected on a single core of an Intel Xeon processor
at 2.10\,GHz (4 logical cores available, benchmarks single-threaded), running
Linux 6.18.5 x86\_64. The implementation was compiled with \texttt{rustc 1.93.1},
release profile (\texttt{opt-level~=~3}, LTO disabled). Criterion 0.5.1 was
used as the benchmarking harness. The \texttt{btv-log} server ran on
\texttt{localhost} to measure socket-loopback submission latency, isolating the
log-submission overhead from network jitter.

\subsection{Metrics}

\begin{enumerate}
  \item \textbf{Submission latency (p50/p99):} end-to-end time of
  \texttt{LogClient::submit\_and\_await}, from sending the verdict hash to
  receiving the signed \texttt{InclusionReceipt}.
  \item \textbf{Memory overhead:} \texttt{std::mem::size\_of::<DeliveryToken>()},
  expected $\approx$\,200--300\,bytes (\texttt{Verdict} 152\,bytes
  + \texttt{Signature} 64\,bytes + \texttt{u64} index).
  \item \textbf{Throughput (ops/s):} decisions per second at steady state under
  simulated concurrent load.
  \item \textbf{Verification cost:} time for an auditor to verify an inclusion
  proof for a log of $N = 10^6$ entries (expected $O(\log N) < 1\,\text{ms}$).
\end{enumerate}

\subsection{Argument for Viability}

The governance overhead of BTV-Transparency is dominated by a single blocking
network round-trip to the Log. In a production AI inference pipeline, this
cost is absorbed by the inference latency itself: a 1--5\,ms log-submission
latency represents at most 10\,\% of the total request cycle for a minimal
LLM classifier (50\,ms), and under 2.5\,\% for a full LLM inference
(200\,ms). The accountability guarantee is therefore practically free at the
scale of any non-trivial model.
